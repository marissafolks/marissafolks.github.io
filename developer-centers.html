Project Background

One source for everything an Android developer needs.

Developer.android.com (DAC) is where Android developers go for the tools they need to build and maintain their applications — from the latest OS requirements to sample code that jumpstarts development.

The initiative behind Developer Centers was to give those developers a centralized location for category-specific content, so they could find exactly what they needed to build their application.

The Ask

Create a space where developers can find everything they need to know about building an Android application. With thousands of tools and libraries available, that information needed to be presented in the most effective way possible.

Problem

Resources were scattered, with little structure to follow.

Documentation, source code, API specifications, blog posts, and videos all lived on DAC — but they were scattered across the site with minimal organizational structure, making it hard for developers to find what they actually needed.

Solution

Market-specific hubs, organized by what developers are building.

Topics were divided into three vertical markets, each giving developers a singular place to find information relevant to the type of app they were building. A consistent layout across all three verticals improved the overall DAC experience.

Process

An agency workflow, driven by the client's research.

The design process here was unconventional — research and planning happened on the client side, and the proposal was brought to Potato to kick off design work based on the Product Requirements Document (PRD).

01

Ticket Created in Taskflow

Taskflow is Google's internal project management system for prioritizing, documenting requirements, and reallocating work based on progress.

  • Each Dev Center request came in as a single ticket, then was broken into smaller, attainable goals to complete across several sprints.
02

Project Kick-off

Once the ticket landed in Taskflow, Potato's Product and Delivery Lead hosted a kickoff with stakeholders, the DAC Product Manager, DAC Designer, and Engineer to gather context, answer questions, and set a launch deadline.

  • Weekly sprint goals and expectations were established in this meeting to move designs forward.
03

Initial Mocks

Mid-to-high-fidelity mocks were designed directly from the PRD. Because DAC relied heavily on its established design system, there was no need to start with low-fidelity wireframes or sketches.

04

Feedback & Iteration

Once initial designs were complete for the sprint, the ticket was updated with the latest Figma file and stakeholders were tagged to review — asynchronously, or in a quick meeting when needed.

  • Any change to delivery requirements was logged in the Taskflow ticket for all stakeholders to see.
  • Requests outside of project scope were revisited and prioritized separately by the project manager.
05

Developer Hand-off

Figma files were refined with notes added for engineering, so the developer could begin building the approved mocks for production.

Design Details

Composable layouts, consistent navigation.

All Dev Centers were built using composable layouts, meaning minimal deviation in overall page structure. Each one starts with an overview page separating high-level process information, then lets developers navigate to documentation, source code, API specs, and videos from there.

Composable layout structure across Dev Centers
  • Navigation — every Dev Center shares the same sub-nav, so developers can always find detailed guides, source code, samples, videos, and blogs in the same place.
  • Hero section — each Dev Center has a unique badge signifying which page a developer is on. Heroes were reserved strictly for informational purposes, never marketing promotions.
  • Core workflow — a dedicated section gives developers a high-level process for planning, designing, developing, and growing their app.
Summary

Composable design made future Dev Centers easier to deliver — giving stakeholders a specific layout to plan and review their content against from day one.
